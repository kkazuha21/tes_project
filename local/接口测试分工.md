> 目标：覆盖系统接口的**正向 + 反向 + 权限**场景；每人负责一组接口模块，产出可复用的接口用例与执行证据（截图/结果）。
>
> 鉴权约定：JWT（`Authorization: Bearer <token>`），接口参考：[SHAREDLAB_API.md](../SHAREDLAB_API.md)

## 成员A：认证与通用鉴权

### 接口清单
- `POST /api/auth/register`
- `POST /api/auth/login`

### 必测点（至少覆盖）
1. 登录成功：返回 `token/type/userId/username/email/role`（字段存在性校验）
2. 登录失败：错误密码/不存在账号 → 401 + 错误提示
3. 注册成功：用户名/邮箱唯一；返回注册成功结构
4. 注册失败：用户名重复、邮箱重复、缺少必填字段 → 400
5. 通用鉴权：
   - 不带 token 调用需要登录接口（任选 2~3 个）→ 401
   - 带无效 token 调用需要登录接口 → 401

### 交付物
- 接口测试用例（建议 12~16 条）
- 执行截图：成功登录拿到 token、失败登录、注册冲突、401 示例

## 成员B：设备与分类接口

### 接口清单
- 设备（公开读 + 管理员写）：
  - `GET /api/devices`（支持 `status`、`categoryId`）
  - `GET /api/devices/{id}`
  - `POST /api/devices`（仅 ADMIN）
  - `PUT /api/devices/{id}`（仅 ADMIN）
  - `DELETE /api/devices/{id}`（仅 ADMIN）
- 分类：
  - `GET /api/categories`
  - `GET /api/categories/{id}`
  - `POST /api/categories`（仅 ADMIN）
  - `PUT /api/categories/{id}`（仅 ADMIN）
  - `DELETE /api/categories/{id}`（仅 ADMIN）

### 必测点（至少覆盖）
1. 设备列表：无筛选/按 `status`/按 `categoryId`（各 1 次）
2. 设备详情：正常 id；不存在 id → 404
3. 管理员写操作：创建→更新→删除的闭环（至少 1 套）
4. 权限：学生/教师调用 `POST/PUT/DELETE /api/devices` → 403
5. 分类：列表/详情正常；不存在 id → 404
6. 分类管理闭环（ADMIN）：新增→更新→删除（至少 1 套）

### 交付物
- 接口测试用例（建议 18~26 条）
- 执行截图：筛选结果、404、403、分类/设备 CRUD 成功

## 成员C：预约调度接口

### 接口清单
- `POST /api/bookings`
- `GET /api/bookings/my`
- `GET /api/bookings?deviceId=...&startTime=...&endTime=...`（设备预约查询）
- `PUT /api/bookings/{id}/cancel`
- `GET /api/bookings/all`（ADMIN/TEACHER）

### 必测点（至少覆盖）
1. 正常创建预约：未来时间，`endTime > startTime`
2. 时间非法：
   - 过去时间 → 400
   - `endTime <= startTime` → 400
3. 冲突检测：同设备重叠时间段 → 409
4. 查询：
   - `my` 列表为空/非空
   - 按设备查询返回时间段
5. 取消预约：成功取消后状态变化（或接口返回字段体现取消结果）
6. 权限：
   - 未登录创建/查询/取消 → 401
   - STUDENT 访问 `GET /api/bookings/all` → 403

### 交付物
- 接口测试用例（建议 16~24 条）
- 执行截图：409 冲突、400 时间校验、取消成功、403/401

## 成员D：报修与用户管理接口

### 接口清单
- 报修（Tickets）：
  - `POST /api/tickets`
  - `GET /api/tickets/my`
  - `GET /api/tickets`（ADMIN/TEACHER）
  - `GET /api/tickets/{id}`
  - `PUT /api/tickets/{id}/status?status=...`（ADMIN/TEACHER）
- 用户管理（仅 ADMIN）：
  - `GET /api/users`（可 `role` 筛选）
  - `GET /api/users/statistics`
  - `GET /api/users/{id}`
  - `POST /api/users`
  - `PUT /api/users/{id}`
  - `PUT /api/users/{id}/toggle-status`
  - `DELETE /api/users/{id}`

### 必测点（至少覆盖）
1. 提交报修：必填字段（`deviceId/title`）校验；成功返回结构
2. 我的报修：空/非空
3. 全部报修：ADMIN/TEACHER 可访问；STUDENT 访问 → 403
4. 状态流转：`SUBMITTED → PROCESSING → CLOSED`（至少 1 条）；非法 status → 400（若后端校验）
5. 报修详情：不存在 id → 404
6. 用户管理：
   - role 筛选（ADMIN/TEACHER/STUDENT 至少各 1 次）
   - 创建用户（含邮箱/用户名冲突的反向用例）
   - toggle-status（启用/禁用）
   - 删除用户（不存在 id → 404 或 204/400 以实现为准，需记录实际行为）
7. 权限：非 ADMIN 调用任何 `/api/users/*` → 403

### 交付物
- 接口测试用例（建议 22~32 条）
- 执行截图：报修状态更新、用户统计、403 权限拒绝、冲突/校验失败

## 统一格式建议（方便最后汇总到 Excel）

每条接口用例建议记录：
- 用例编号（如 API-AUTH-001）
- 接口/方法
- 前置条件（角色、token）
- 请求参数/Body
- 预期结果（状态码 + 关键字段/关键字断言）
- 实测结果（状态码、响应摘要）
- 结论（通过/不通过）
- 测试人员/测试时间
