# SharedLab 项目用例描述

> 范围：用户权限模块、设备管理模块、预约调度模块、使用记录模块、故障报修模块。
>
> 参与者定义：
>
> - 访客：未登录用户
> - 学生（STUDENT）
> - 教师（TEACHER）
> - 管理员（ADMIN）
>
> 鉴权约定：登录后通过 JWT（`Authorization: Bearer <token>`）访问需要登录的接口；管理员/教师的管理能力受后端 `@PreAuthorize` 等权限限制。

## 一、用户权限模块

### UC-01 用户注册

- 用例名称：用户注册
- 用例描述：用户创建账户，系统保存用户基本信息并为其分配角色（默认学生）。
- 参与者：访客（主参与者）
- 前置条件：
  - 用户未登录
  - 用户名、邮箱尚未被占用
- 后置条件：
  - 成功：用户记录创建完成，可使用账号登录
  - 失败：不产生任何用户记录
- 基本事件流：
  1. 访客进入注册入口，填写用户名、邮箱、密码。
  2. 系统校验必填项与格式（用户名长度、邮箱格式、密码长度等）。
  3. 系统保存用户信息（密码进行加密存储），默认角色为 STUDENT（若允许选择则按选择保存）。
  4. 系统返回“注册成功”结果。
- 备选事件流：
  - A1：用户名或邮箱已存在 → 系统返回失败提示，用户可修改后重试。
  - A2：参数不合法（缺失/长度不符合）→ 系统提示校验错误，不创建用户。

### UC-02 用户登录

- 用例名称：用户登录
- 用例描述：用户使用用户名+密码登录，系统签发 JWT，并根据角色引导进入对应首页。
- 参与者：学生/教师/管理员（主参与者）
- 前置条件：
  - 用户已注册
  - 用户状态为启用（status=1）
- 后置条件：
  - 成功：客户端获得 JWT 并保存会话信息（userId/role 等），可访问受保护资源
  - 失败：不建立会话
- 基本事件流：
  1. 用户输入用户名与密码并提交。
  2. 系统校验账号密码正确性。
  3. 系统生成 JWT，并返回 token、用户信息与角色。
  4. 前端保存 token，后续请求自动附加 `Authorization` 请求头。
  5. 前端根据角色跳转到对应页面：
     - ADMIN → 管理后台首页
     - TEACHER → 教师端首页
     - STUDENT → 学生端首页
- 备选事件流：
  - A1：用户名或密码错误 → 返回 401/错误提示，停留在登录页。
  - A2：用户被禁用（status=0）→ 返回错误提示（或拒绝登录）。
  - A3：token 过期/无效 → 访问受保护接口返回 401，前端清理会话并跳转登录。

### UC-03 角色访问控制（前端路由守卫 + 后端接口鉴权）

- 用例名称：角色访问控制
- 用例描述：系统限制不同角色访问不同页面和接口，避免越权操作。
- 参与者：访客、学生、教师、管理员
- 前置条件：
  - 已配置路由守卫与后端安全策略
- 后置条件：
  - 用户只能访问其权限范围内的页面/接口
- 基本事件流：
  1. 用户尝试访问某页面或调用某 API。
  2. 前端路由守卫检查是否登录以及角色是否匹配。
  3. 后端对受保护接口校验 JWT 与角色权限。
  4. 通过校验则放行访问，返回正常结果。
- 备选事件流：
  - A1：访客访问受保护页面 → 前端重定向至登录页。
  - A2：已登录但访问非本角色页面 → 前端重定向到本角色首页。
  - A3：调用管理员/教师限定接口但权限不足 → 后端返回 403。

### UC-04 管理员用户管理

- 用例名称：管理员用户管理（查询/新增/修改/启用禁用/删除）
- 用例描述：管理员对系统用户进行维护，包括按角色筛选查询、创建用户、修改用户、启用/禁用用户、删除用户。
- 参与者：管理员（主参与者）
- 前置条件：
  - 管理员已登录且 token 有效
- 后置条件：
  - 成功：用户数据按操作被新增/更新/删除，或状态被切换
  - 失败：数据不变并给出错误信息
- 基本事件流：
  1. 管理员进入用户管理页面。
  2. 系统展示用户列表（可选：按角色筛选）。
  3. 管理员执行以下一种操作：
     - 新增用户：填写用户名、邮箱、角色、初始密码 → 提交 → 系统创建用户。
     - 修改用户：编辑用户信息（可选重置密码）→ 提交 → 系统保存变更。
     - 启用/禁用：对指定用户切换状态 → 系统更新 status。
     - 删除用户：确认删除 → 系统删除用户。
  4. 系统返回成功结果并刷新列表。
- 备选事件流：
  - A1：用户名/邮箱冲突 → 返回错误提示，管理员修改后重试。
  - A2：参数校验失败 → 返回 400 与字段级错误信息。
  - A3：权限不足或 token 失效 → 返回 403/401，前端引导重新登录。

## 二、设备管理模块

### UC-05 查看设备列表（含筛选）

- 用例名称：查看设备列表
- 用例描述：用户查看设备清单，并可按状态、分类筛选。
- 参与者：访客/学生/教师/管理员（主参与者）
- 前置条件：无（设备列表/详情通常允许公开访问；如部署策略调整为需登录，则以实际安全配置为准）
- 后置条件：
  - 返回设备列表数据
- 基本事件流：
  1. 用户进入设备列表页。
  2. 系统请求设备列表（可带 status、categoryId 等筛选条件）。
  3. 系统返回设备列表并展示。
- 备选事件流：
  - A1：筛选条件不合法 → 系统提示并回退到默认列表。
  - A2：后端不可用/网络异常 → 页面提示加载失败，可重试。

### UC-06 查看设备详情

- 用例名称：查看设备详情
- 用例描述：用户查看单个设备的详细信息（名称、状态、位置、描述、分类等）。
- 参与者：访客/学生/教师/管理员（主参与者）
- 前置条件：设备存在
- 后置条件：
  - 返回设备详情数据
- 基本事件流：
  1. 用户在设备列表中点击某设备。
  2. 系统请求设备详情并展示。
- 备选事件流：
  - A1：设备不存在 → 返回 404，页面提示“设备不存在”。

### UC-07 管理员维护设备（新增/编辑/删除）

- 用例名称：管理员维护设备
- 用例描述：管理员对设备进行新增、修改、删除，并维护设备状态与分类关联。
- 参与者：管理员（主参与者）
- 前置条件：
  - 管理员已登录且 token 有效
- 后置条件：
  - 成功：设备数据被新增/更新/删除
- 基本事件流：
  1. 管理员进入设备管理页面。
  2. 系统展示设备列表。
  3. 管理员执行以下一种操作：
     - 新增设备：填写名称、状态、位置、描述、分类 → 提交 → 创建成功。
     - 编辑设备：修改字段/状态/分类 → 提交 → 更新成功。
     - 删除设备：确认删除 → 系统删除设备。
  4. 系统刷新列表显示最新结果。
- 备选事件流：
  - A1：必填字段缺失/格式不合法 → 返回 400，提示校验错误。
  - A2：权限不足 → 返回 403。
  - A3：删除时存在业务约束（例如已有预约/报修引用）→ 返回失败并提示原因（以实际实现为准）。

### UC-08 管理员维护设备分类（新增/编辑/删除）

- 用例名称：管理员维护设备分类
- 用例描述：管理员维护设备分类，用于设备归类与筛选。
- 参与者：管理员（主参与者）
- 前置条件：管理员已登录
- 后置条件：
  - 成功：分类数据被新增/更新/删除
- 基本事件流：
  1. 管理员进入分类管理页面。
  2. 系统加载分类列表（可显示每类设备数量）。
  3. 管理员新增/编辑/删除分类并提交。
  4. 系统返回成功并刷新列表。
- 备选事件流：
  - A1：分类名称重复 → 返回错误提示。
  - A2：删除分类时仍有关联设备 → 删除失败并提示（以实际实现为准）。

## 三、预约调度模块

### UC-09 查看设备已预约时段（用于日历展示）

- 用例名称：查看设备已预约时段
- 用例描述：用户在设备详情页查看该设备已有预约时间段，辅助选择空闲时段。
- 参与者：学生/教师/管理员（主参与者）
- 前置条件：
  - 用户已登录（如接口放行策略为公开则可省略）
  - 设备存在
- 后置条件：
  - 返回该设备预约列表（可选：按时间范围查询）
- 基本事件流：
  1. 用户进入设备详情页。
  2. 系统请求该设备的预约列表（用于日历/列表展示）。
  3. 系统返回预约时段并展示。
- 备选事件流：
  - A1：设备不存在 → 返回 404。
  - A2：无预约记录 → 返回空数组，页面显示“暂无预约”。

### UC-10 创建预约（含冲突检测）

- 用例名称：创建设备预约
- 用例描述：用户选择设备与时间段发起预约，系统进行时间冲突检测，通过则创建预约。
- 参与者：学生/教师（主参与者）
- 前置条件：
  - 用户已登录且 token 有效
  - startTime、endTime 为未来时间，且 endTime > startTime
  - 设备存在
- 后置条件：
  - 成功：生成一条预约记录（通常状态为 CONFIRMED）
  - 失败：不产生预约记录
- 基本事件流：
  1. 用户在设备详情页选择开始时间与结束时间并提交。
  2. 系统校验时间合法性。
  3. 系统执行冲突检测：判断是否与现有预约时间段重叠（排除已取消状态）。
  4. 无冲突则创建预约记录并返回预约信息。
  5. 前端提示“预约成功”，并刷新“已有预约时段/我的预约”。
- 备选事件流：
  - A1：时间冲突 → 系统返回 409，并提示“该时间段已被预约”。
  - A2：时间非法（过去时间/结束早于开始）→ 返回 400 并提示。
  - A3：设备不存在 → 返回 404。
  - A4：未登录/登录过期 → 返回 401，前端跳转登录。

### UC-11 取消预约

- 用例名称：取消预约
- 用例描述：用户取消自己创建的预约，系统将预约状态更新为 CANCELLED（或等价状态）。
- 参与者：学生/教师（主参与者）
- 前置条件：
  - 用户已登录
  - 预约存在且属于当前用户（权限规则以实际实现为准）
- 后置条件：
  - 成功：预约状态变更为已取消
- 基本事件流：
  1. 用户在“我的预约”列表选择一条预约点击取消。
  2. 系统校验权限与预约状态。
  3. 系统更新预约状态并返回更新后的预约信息。
  4. 前端刷新列表。
- 备选事件流：
  - A1：预约不存在 → 404。
  - A2：无权取消他人预约 → 403。
  - A3：预约已开始/已结束不允许取消 → 返回失败提示（以实际实现为准）。

### UC-12 查看我的预约

- 用例名称：查看我的预约
- 用例描述：用户查看自己创建的预约记录与状态。
- 参与者：学生/教师（主参与者）
- 前置条件：用户已登录
- 后置条件：返回当前用户的预约列表
- 基本事件流：
  1. 用户进入“我的预约”。
  2. 系统请求并返回该用户的预约列表。
  3. 前端展示预约记录与状态。
- 备选事件流：
  - A1：无记录 → 返回空列表并提示“暂无预约”。

### UC-13 查看全部预约（教师/管理员）

- 用例名称：查看全部预约
- 用例描述：教师或管理员查看系统内所有预约记录，用于管理与排班查看。
- 参与者：教师、管理员（主参与者）
- 前置条件：教师/管理员已登录
- 后置条件：返回全部预约列表
- 基本事件流：
  1. 教师/管理员进入预约管理页面。
  2. 系统请求全部预约列表并展示。
- 备选事件流：
  - A1：权限不足 → 403。

## 四、使用记录模块

> 说明：当前项目中“使用记录”主要通过“预约记录（bookings）”体现；即设备的使用时间段与使用人信息可通过预约列表追溯。

### UC-14 查看设备使用记录（按设备追溯预约记录）

- 用例名称：查看设备使用记录
- 用例描述：用户查看某设备在一段时间内的预约记录，用于了解设备占用情况与使用历史。
- 参与者：学生/教师/管理员（主参与者）
- 前置条件：
  - 用户已登录（如接口策略为公开则可省略）
  - 设备存在
- 后置条件：
  - 返回该设备的预约记录（可选：按 startTime/endTime 范围）
- 基本事件流：
  1. 用户进入设备详情或使用记录入口。
  2. 系统按设备 ID 查询预约记录（可选时间范围）。
  3. 系统返回记录并展示时间段列表。
- 备选事件流：
  - A1：无记录 → 返回空列表。
  - A2：参数不合法（时间范围格式错误）→ 返回 400。

### UC-15 查看全量使用记录（教师/管理员）

- 用例名称：查看全量使用记录
- 用例描述：教师/管理员从全局视角查看系统所有预约记录，作为使用统计与排班参考。
- 参与者：教师、管理员（主参与者）
- 前置条件：教师/管理员已登录
- 后置条件：返回全部预约记录
- 基本事件流：
  1. 教师/管理员打开“预约管理/使用记录”页面。
  2. 系统请求全部预约列表并展示。
- 备选事件流：
  - A1：权限不足 → 403。

## 五、故障报修模块

### UC-16 提交报修单

- 用例名称：提交报修单
- 用例描述：用户针对某设备提交故障/问题描述，生成报修工单。
- 参与者：学生/教师（主参与者）
- 前置条件：
  - 用户已登录
  - 设备存在
- 后置条件：
  - 成功：生成一条报修单记录（初始状态通常为 SUBMITTED）
- 基本事件流：
  1. 用户进入设备详情或报修入口。
  2. 选择设备，填写标题、内容（可选）、严重程度（可选）。
  3. 提交报修。
  4. 系统创建报修单并返回详情。
  5. 前端提示成功，并在“我的报修”中可见。
- 备选事件流：
  - A1：必填字段缺失（如 title）→ 400。
  - A2：设备不存在 → 404。
  - A3：未登录/登录过期 → 401。

### UC-17 查看我的报修单

- 用例名称：查看我的报修单
- 用例描述：用户查看自己提交的报修单列表与状态变化。
- 参与者：学生/教师（主参与者）
- 前置条件：用户已登录
- 后置条件：返回当前用户的报修单列表
- 基本事件流：
  1. 用户进入“我的报修”。
  2. 系统返回该用户的报修单列表。
  3. 前端展示标题、设备、状态、创建时间等。
- 备选事件流：
  - A1：无记录 → 返回空列表。

### UC-18 教师/管理员处理报修（查看列表/详情/更新状态）

- 用例名称：处理报修单
- 用例描述：教师或管理员查看全量报修单，进入详情并更新处理状态（如 PROCESSING/CLOSED/REJECTED）。
- 参与者：教师、管理员（主参与者）
- 前置条件：教师/管理员已登录
- 后置条件：
  - 成功：报修单状态被更新
- 基本事件流：
  1. 教师/管理员进入报修管理页面。
  2. 系统加载全部报修单列表。
  3. 选择某报修单进入详情。
  4. 选择新的状态并提交。
  5. 系统更新报修单状态并返回更新后的详情。
- 备选事件流：
  - A1：权限不足 → 403。
  - A2：报修单不存在 → 404。
  - A3：状态值不在允许范围内 → 400。

### UC-19 查看报修处理记录（维修日志）

- 用例名称：查看报修处理记录
- 用例描述：在报修单详情中查看处理过程记录（maintenance_logs），用于追溯“做了什么/何时做的”。
- 参与者：学生/教师/管理员（主参与者）
- 前置条件：
  - 用户已登录
  - 报修单存在且用户有权查看（本人/管理端）
- 后置条件：
  - 展示报修单关联的处理记录列表
- 基本事件流：
  1. 用户打开某报修单详情。
  2. 系统返回报修单信息以及关联的维修日志列表（如有）。
  3. 前端展示日志条目（action、note、created_at 等）。
- 备选事件流：
  - A1：无维修日志 → 显示“暂无处理记录”。
  - A2：无权查看他人报修单 → 403。
