### (1) UC001-1 登录、注册

**功能目标**：为不同角色（学生/教师/管理员）提供账号注册与登录入口，登录成功后获得身份凭证（如 Token）并进入对应角色的功能页面。

**实现过程（建议按步骤写成流程图/活动图）**
1. 用户打开登录/注册页面。
2. 选择“登录”或“注册”。
3. 注册流程：
   - 用户填写用户名/邮箱/密码等必要字段。
   - 前端进行基础校验（必填、格式、密码长度等）。
   - 调用后端注册接口提交数据。
   - 后端进行合法性校验与唯一性检查（如邮箱是否已存在）。
   - 校验通过后写入用户数据（默认角色/默认状态），返回“注册成功”。
4. 登录流程：
   - 用户输入账号（邮箱/用户名）与密码。
   - 前端调用后端登录接口。
   - 后端校验账号与密码是否匹配、账号状态是否可用。
   - 登录成功后生成身份凭证（如 JWT Token）并返回。
5. 前端保存身份凭证，并拉取用户信息（角色、姓名等）。
6. 前端根据角色跳转到对应的首页/控制台（管理员/教师/学生）。

**异常/备选**
- 输入为空或格式不正确：前端提示并阻止提交。
- 账号已存在：后端返回错误信息，前端提示。
- 密码错误/账号不存在：提示“账号或密码错误”。
- 账号被禁用：提示“账号不可用/已停用”。

### UC001-2 用户管理

**功能目标**：提供用户信息的管理与维护能力（查看、编辑、启用/禁用、删除等由系统实现的管理动作），并支撑用户侧个人资料维护（见 UC001-2-1/2/3）。

**实现过程**
1. 用户登录后，系统根据角色显示对应的“用户管理/个人中心”入口。
2. 普通用户（学生/教师）主要进行个人信息维护（见 UC001-2-1/2/3）。
3. 管理员进入用户管理页面：
   - 拉取用户列表。
   - 支持对单个用户执行管理操作（如编辑、启用/禁用、删除）。
4. 后端对管理操作做权限校验（仅管理员可执行管理端动作）。
5. 操作成功后刷新列表或局部更新页面数据。

**异常/备选**
- 权限不足：后端返回 403，前端提示“权限不足”。
- 目标用户不存在：提示“用户不存在”。

### (2) UC001-2-1 修改个人信息

**功能目标**：用户更新自己的基础资料（如昵称、姓名等个人信息字段）。

**实现过程**
1. 用户登录后进入“个人中心/我的信息”。
2. 系统展示当前个人信息（从后端读取）。
3. 用户编辑可修改字段。
4. 前端做基础校验（必填/长度）。
5. 调用后端“更新个人信息”接口提交变更。
6. 后端校验 Token 身份，确保仅能修改本人信息。
7. 后端更新数据库并返回成功。
8. 前端提示成功，并刷新页面展示最新信息。

**异常/备选**
- 未登录或 Token 失效：跳转登录。
- 字段不合法：提示具体原因。

### (3) UC001-2-2 修改密码

**功能目标**：用户安全地修改登录密码。

**实现过程**
1. 用户进入“修改密码”页面。
2. 输入旧密码、新密码、确认新密码。
3. 前端校验：新密码规则、两次输入一致。
4. 调用后端“修改密码”接口。
5. 后端验证旧密码是否正确。
6. 验证通过后，对新密码进行加密/哈希处理并更新数据库。
7. 返回成功后，前端提示“修改成功”，可要求重新登录（可选，按系统实现）。

**异常/备选**
- 旧密码错误：提示“旧密码不正确”。
- 新密码不符合规则：提示规则要求。

### (4) UC001-2-3 修改邮箱信息

**功能目标**：用户更新邮箱信息，用于账号联系或登录标识。

**实现过程**
1. 用户进入“修改邮箱”页面。
2. 输入新邮箱。
3. 前端校验邮箱格式。
4. 调用后端“更新邮箱”接口。
5. 后端校验身份（只能改本人），并检查新邮箱是否已被占用。
6. 通过后更新数据库并返回成功。
7. 前端提示成功，刷新页面。

**异常/备选**
- 邮箱已存在：提示“邮箱已被占用”。
- 邮箱格式错误：前端提示。

### (1) UC002-1 查看设备

**功能目标**：展示设备列表，包含设备名称、状态、位置等关键信息。

**实现过程**
1. 用户进入设备列表页面。
2. 前端向后端发起“获取设备列表”请求，可携带筛选参数（如状态、分类）。
3. 后端从数据库查询设备数据，必要时按筛选条件过滤。
4. 返回设备数组（每条包含 id、name、status、location 等）。
5. 前端以列表/卡片形式渲染设备信息。

**异常/备选**
- 返回空数组：前端展示“暂无设备”。
- 服务异常：前端提示“获取设备列表失败”。

### UC002-2 管理设备

**功能目标**：管理员对设备信息进行维护，包括新增/编辑/删除设备，以及维护设备分类（见 UC002-2-2）。

**实现过程**
1. 管理员登录后进入“设备管理”页面。
2. 系统展示设备列表与管理入口（新增、编辑、删除等）。
3. 所有管理接口由后端进行权限校验（仅管理员可操作）。
4. 设备数据更新后刷新列表，确保界面与数据库一致。

### (2) UC002-1-1 按设备分类筛选

**功能目标**：用户按分类查看设备列表，减少查找成本。

**实现过程**
1. 设备列表页面加载分类下拉框（从后端读取分类列表）。
2. 用户选择某分类。
3. 前端携带 `categoryId` 请求设备列表接口。
4. 后端按分类条件查询设备并返回。
5. 前端刷新设备列表显示筛选结果。

**异常/备选**
- 分类不存在/无设备：返回空数组，前端显示“暂无设备”。

### (3) UC002-1-2 查看设备详情

**功能目标**：查看单台设备的详细信息（名称、状态、位置、描述、所属分类等）。

**实现过程**
1. 用户在设备列表点击某设备卡片/行。
2. 前端跳转到设备详情页（携带设备 id）。
3. 前端调用后端“获取设备详情”接口（按 id 查询）。
4. 后端返回设备完整信息。
5. 前端展示详情；必要时显示可执行操作入口（如管理员编辑、用户预约等由系统实现）。

**异常/备选**
- 设备不存在：提示“设备不存在”。

### (4) UC002-2-1 编辑设备

**功能目标**：管理员编辑设备基础信息与状态。

**实现过程**
1. 管理员在设备管理列表选择“编辑”。
2. 前端弹出编辑表单（名称、状态、位置、描述、分类等）。
3. 前端校验必填字段。
4. 调用后端“更新设备”接口提交修改。
5. 后端进行权限校验与字段校验。
6. 更新数据库并返回成功。
7. 前端关闭弹窗，刷新设备列表。

**异常/备选**
- 权限不足：提示“权限不足”。
- 参数不合法：提示具体原因。

### (5) UC002-2-2 管理设备分类

**功能目标**：管理员维护设备分类（新增/编辑/删除分类，或在设备编辑中关联分类）。

**实现过程**
1. 管理员进入“设备分类”管理页面。
2. 系统加载分类列表。
3. 管理员可新增分类：填写分类名称/描述并提交。
4. 管理员可编辑分类：修改名称/描述并提交。
5. 管理员可删除分类：系统提示确认；删除前需考虑是否存在关联设备（按系统规则处理：禁止删除/先解绑/级联等）。
6. 操作成功后刷新分类列表。

**异常/备选**
- 分类被设备引用且不允许删除：提示“分类下仍有设备，无法删除”。

### (1) UC003-1 预约

**功能目标**：对设备使用时间进行预约与调度，避免冲突。

**实现过程**
1. 用户在设备详情或预约页面选择设备与预约时间段。
2. 前端提交预约请求到后端。
3. 后端校验：用户身份、设备是否存在、时间段合法。
4. 后端进行冲突检测：同设备在同一时间段是否已有有效预约。
5. 若无冲突，创建预约记录并设置初始状态（如待确认/已确认等按系统实现）。
6. 返回成功后前端提示并刷新预约列表。

**异常/备选**
- 时间冲突：提示“该时段已被预约”。
- 设备不可用（维护/已占用等）：提示“设备当前不可预约”。

### (2) UC003-1-1 创建预约

**功能目标**：提交一条新的预约申请。

**实现过程**
1. 选择设备。
2. 选择开始时间/结束时间。
3. 填写预约用途（如需要）。
4. 前端校验时间：结束时间晚于开始时间、不能早于当前时间（按系统规则）。
5. 调用后端创建预约接口。
6. 后端校验并写入预约表。
7. 返回成功，前端展示新预约信息。

### (3) UC003-1-2 取消预约

**功能目标**：用户或管理员取消尚未开始/允许取消的预约。

**实现过程**
1. 用户在“我的预约/预约管理”中选择一条预约。
2. 点击“取消”，前端弹出确认。
3. 调用后端取消预约接口。
4. 后端校验：
   - 身份：本人或管理员。
   - 状态：是否允许取消（例如已结束不可取消）。
5. 更新预约状态为“已取消”。
6. 返回成功，前端刷新列表。

**异常/备选**
- 不允许取消：提示“预约已开始/已结束，无法取消”。

### (4) UC003-1-3 查看预约

**功能目标**：查看预约列表与详情（按角色可见范围不同）。

**实现过程**
1. 用户进入预约列表页面。
2. 前端请求后端预约列表接口。
3. 后端按角色返回数据：
   - 学生/教师：默认只返回自己的预约。
   - 管理员：可查看全量预约或按条件筛选。
4. 前端渲染列表，可点击进入预约详情。

**异常/备选**
- 无数据：展示“暂无预约”。

### (1) UC005-1 报修故障

**功能目标**：学生/教师在发现设备故障时提交报修单，并跟踪处理状态。

**实现过程**
1. 用户在设备详情或报修入口点击“提交报修”。
2. 填写报修内容：设备、故障描述、（可选）图片/附件等（按系统实现）。
3. 前端校验必填项。
4. 调用后端创建报修单接口。
5. 后端校验身份与设备存在性，写入报修单并设置初始状态（如待处理）。
6. 返回成功后前端提示并可跳转到“我的报修单”。

### UC005-2 管理报修

**功能目标**：管理员/教师对报修单进行处理、记录处理过程，并形成处理记录。

**实现过程**
1. 管理员/教师进入报修管理列表。
2. 系统加载待处理/处理中/已完成的报修单。
3. 选择某条报修单查看详情并执行处理动作（见 UC005-2-1）。
4. 处理过程中记录处理结果与备注（见 UC005-2-2）。

### (2) UC005-1-1 提交报修单

**功能目标**：创建一条新的报修单。

**实现过程**
1. 选择设备（或从设备详情自动带入）。
2. 输入故障描述、发生位置/时间等信息（按系统实现）。
3. 提交到后端。
4. 后端生成报修单编号/创建时间，保存并返回成功。

**异常/备选**
- 未登录：提示登录。
- 设备不存在：提示“设备不存在”。

### (3) UC005-1-2 查看我的报修单

**功能目标**：提交人查看自己提交的报修单列表与状态。

**实现过程**
1. 用户进入“我的报修单”。
2. 前端请求后端“按当前用户查询报修单”接口。
3. 后端根据 Token 获取用户 id，仅返回该用户创建的报修单。
4. 前端展示列表，可进入详情查看处理进展。

### (4) UC005-2-1 处理报修（管理员、教师）

**功能目标**：对报修单进行受理、处理中、完成等状态流转，并填写处理结果。

**实现过程**
1. 管理员/教师打开某条报修单详情。
2. 点击“开始处理/更新状态/完成”。
3. 填写处理说明、处理结果。
4. 前端调用后端更新报修单状态接口。
5. 后端校验角色权限，更新报修单状态与处理信息。
6. 返回成功，前端刷新详情与列表。

**异常/备选**
- 权限不足：提示“权限不足”。
- 状态流转非法（例如已完成再处理）：提示“状态不允许”。

### (5) UC005-2-2 查看报修处理记录

**功能目标**：查看报修单的处理历史（谁在什么时候做了什么处理）。

**实现过程**
1. 在报修单详情页展示“处理记录”区域。
2. 前端请求后端“获取处理记录”接口（按报修单 id）。
3. 后端返回处理记录列表（时间、处理人、动作、备注等）。
4. 前端按时间倒序展示，便于追踪。

**异常/备选**
- 无记录：展示“暂无处理记录”。
